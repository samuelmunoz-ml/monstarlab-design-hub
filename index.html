The Handoff Process

Standard operating procedures for moving designs from Figma to production — eliminating ambiguity between design intent and the final built product.

File Nomenclature & Organisation

A clean, predictable file structure is the foundation of an effective handoff. When engineers open a file, they should instantly understand its purpose, state, and contents — without asking anyone.

Naming Convention

[PROJECT-CODE]_[FEATURE]_[TYPE]_v[VERSION]_[STATUS] e.g. SHR_LoyaltyFlow_UI_v3_HANDOFF.fig  ·  USH_Checkout_UX_v1_REVIEW.fig
TokenDescriptionExample Values
[PROJECT-CODE]2–4 letter client/project abbreviationSHR, USH, MLI
[FEATURE]Feature or module name, PascalCaseLoyaltyFlow, Checkout, Onboarding
[TYPE]File type descriptorUX, UI, Prototype, Assets
[VERSION]Numeric version, zero-paddedv1, v2, v12
[STATUS]Current lifecycle stateWIP, REVIEW, FINAL, HANDOFF

Definition of Done (DoD)

A design is not done when it looks finished. It is done when it meets every criterion on this checklist. Nothing moves to the Ready for Dev page without passing this gate.

🖼
DoD Checklist — Figma Template
Screenshot of the DoD checklist component as it appears pasted on a Figma frame
A — 7 items
Coverage & Completeness
  • All primary user flows are fully designed, end-to-end
  • All interactive states present: Default, Hover, Focus, Active, Disabled, Loading
  • All form states covered: Empty, Filled, Validation Error, Success
  • Empty states designed for all data-dependent views
  • Error states designed with specific, actionable copy
  • Responsive variants present for all defined breakpoints (Mobile, Tablet, Desktop)
  • Edge cases documented: long text strings, missing images, maximum data sets
B — 5 items
Design System Compliance
  • All components are instances from the approved ML Design System library — no detached overrides
  • All colour values map to defined design tokens (no direct hex values on layers)
  • All typography uses defined text styles
  • All spacing follows the defined grid and spacing scale
  • All icons are from the approved icon library at the correct weight
C — 6 items
Accessibility
  • Colour contrast meets WCAG AA (4.5:1 for normal text, 3:1 for large text and UI components)
  • All interactive elements have visible focus states designed
  • Touch targets meet minimum size requirements (44×44px minimum)
  • All images have alt text or a note indicating decorative intent
  • Reading order is logical and verified via layer panel top-to-bottom review
  • No information is conveyed by colour alone — icons, labels, or patterns supplement
D — 5 items
File Hygiene
  • All layers in the ready-for-dev page are named per Section 01 conventions
  • Auto Layout applied to all frames used as developer reference
  • No hidden layers in the ready-for-dev page
  • All exported assets included in the designated Exports frame
  • No broken component links in the file
E — 4 items
Annotation & Documentation
  • All non-obvious interactions annotated with the Annotation component
  • Transition timings and easing curves specified for animated elements
  • Any bespoke or non-standard CSS properties called out explicitly
  • A flow description note added at the top of each flow group

Technical Specs & Annotations

Specs bridge the gap between design intent and implementation. Every value an engineer needs must be annotated — if it isn't documented, it doesn't exist.

🖼
Annotation Example — Spacing & Typography Redlines
Screenshot showing a Figma frame with spacing tokens, type styles, and colour tokens annotated

Tokens to Annotate

Spacing Tokens
Use the ML spacing scale (4, 8, 12, 16, 24, 32, 48, 64px). Never annotate arbitrary pixel values — always map to the nearest token and note if a deviation is intentional.
Typography Tokens
Annotate with the text style name (e.g. body-md-regular), not raw values. If a custom size is required, flag it explicitly with a reason.
Colour Tokens
Always reference token names (e.g. surface/primary, text/secondary), not hex values. Direct hex values in annotations indicate a token gap that must be resolved before handoff.
Redlines
Use the ML Annotation component for redlines. Call out padding, margin, gap, and border-radius values explicitly. Annotate the component's responsive behaviour at each breakpoint.

Transition Timing Tokens

All animated transitions must be documented using these standard tokens. Format annotations as: transition: [property] [duration]ms [easing-token]

TokenDurationEasingUse Case
transition-instant0ms—Focus ring appearance, state changes where animation would feel janky
transition-fast100msease-outMicro-interactions: hover highlights, button presses, icon state changes
transition-base200msease-in-outStandard UI transitions: colour fades, opacity changes, panel reveals
transition-moderate300msease-in-outModal open/close, dropdown appear, tab panel switch
transition-slow500msease-in-outPage-level transitions, onboarding flows, complex reveals
transition-enter400mscubic-bezier(0.22, 1, 0.36, 1)Elements entering the viewport — slight spring feel
transition-exit250msease-inElements leaving the viewport — faster exit maintains perceived snappiness
★
Pro Tip: When in doubt, be faster. User research consistently shows transitions above 400ms feel sluggish on standard UI elements. Reserve longer durations for motion that communicates meaning — not decoration.

Figma File Hygiene

The structural quality of a Figma file directly determines how efficiently an engineer can inspect, understand, and implement a design. A clean file is a professional act of respect for the people who build what you design.

Auto Layout Rules

Auto Layout is mandatory for all frames intended for developer handoff. It communicates the intended responsive behaviour of a component, not just its static appearance.

Hug Contents
This element should wrap its content — no fixed width/height in the component. Maps to width: fit-content.
Fill Container
This element should be 100% of its parent. Maps to width: 100% or flex: 1.
Fixed
This element has a defined dimension that does not change with content. Maps to an explicit width or height value.
Gap Values
Must reference a spacing token. Padding must be set uniformly or with explicit horizontal/vertical overrides — never set by eye.

Component & Variant Rules

  • Use the ML Design System library for all standard UI elements — buttons, inputs, badges, modals, navigation
  • If a required component doesn't exist, raise a DS request in #design-system before designing a local version
  • Never detach a component to make a visual change — use variant overrides or nested component swaps
  • Variant property names: Title Case, no spaces (e.g. Size, State, Type). Values: lowercase with hyphens (e.g. default, hover, error)
  • Use Figma's Component Property panel — not frame name hacks — for variant values

Inspect-Friendly Practices

  • Turn off "Clip content" on frames that expand with content unless overflow: hidden is intentional
  • Mark all placeholder images as "Fill" with a note specifying the intended aspect ratio and minimum size
  • Remove all unnecessary prototype connections from the ready-for-dev page
  • Add a component description in the Figma component panel for every DS component
  • Zero hidden layers in the ready-for-dev page — hiding is not the same as deleting
★
Pro Tip: Once per sprint, run a quick "Engineer's Review" — open the Inspect panel and try to read your own file as if you're the engineer seeing it for the first time. You'll find annotation gaps, missing states, and ambiguous layout relationships that are invisible from the design view.

Figma Comment Protocol

Start every comment with one of these tags, then @mention the relevant person and include enough context that someone reading it three months later will understand it.

[QUESTION] [DECISION] [NOTE] [BUG] [BLOCKED]
  • Resolve comments only when the issue is fully addressed — not when you've replied
  • All comments on a ready-for-dev frame must be resolved before engineers begin implementation
  • Do not use Figma comments for social conversation — use Slack for that
  • Comment threads older than two weeks with no resolution must be escalated to the PM

Handling Design Debt

Design debt is any design decision that is knowingly incomplete, inconsistent, or non-ideal — shipped because of time, scope, or technical constraints. It must be acknowledged, documented, and scheduled — not ignored.

PriorityDefinitionTarget Resolution
P1 — CriticalAccessibility failure or user-facing error state with no resolution pathWithin current sprint
P2 — HighMissing key user flow state or significant inconsistency visible to end usersWithin next two sprints
P3 — LowCosmetic inconsistency, token deviation, or incomplete annotationQuarterly DS review

Debt protocol: Identify → log a Jira ticket with the design-debt label → Annotate the Figma frame with a link to the ticket → Triage bi-weekly with the Design Lead and PM → Resolve and update the Cover page version history.

★
Pro Tip: Never let debt accumulate beyond 15 open tickets without a triage session. When design debt exceeds this threshold, it becomes invisible to the team — no one knows which debt is blocking, which is cosmetic, and which has already been informally resolved. Visibility is discipline.

The Async-First Protocol

Every tool in our stack has a job. Using the right tool eliminates unnecessary meetings and creates a searchable record of decisions.

💬 Slack
Use for: Quick questions, status pings, sharing links, async standups, team announcements.
Not for: lengthy design feedback, file reviews, or decisions that need a paper trail. Threads are acceptable for short back-and-forth (≤ 5 replies). Escalate longer discussions to a Doc or a Loom.
🎥 Loom
Use for: Walkthroughs of complex flows, onboarding demos, QA review recordings, presenting work to async stakeholders.
Default to Loom before scheduling a meeting. A 5-minute Loom replaces a 30-minute sync for 90% of design review scenarios. Include a summary in the Slack post — don't make people watch to understand context.
✏️ Figma Comments
Use for: Contextual feedback tied directly to design elements, review rounds, design QA, stakeholder sign-off.
Tag people explicitly with @mention. Always resolve a comment with a written outcome — don't just close it. Use the tagging convention: [QUESTION], [DECISION], [FEEDBACK], [ACTION]. Never leave an unresolved comment in a file that's READY FOR DEV.
🎯 Jira
Use for: All task tracking, sprint planning, bug reports, design debt logging, and delivery milestones.
Figma file links belong in Jira, not in Slack messages. Status updates happen in Jira — not in a DM. If someone asks "where is this?" the answer should always be findable in the ticket.
📄 Google Docs
Use for: Briefs, specifications, retro notes, decision logs, research synthesis, any written communication that needs structured review.
Decisions made in meetings must be documented in a shared Doc within 24 hours. Enable "suggestion mode" for review cycles. Never send a doc for review over DM — share it in the relevant Slack channel with context.

When to call a meeting

✓ Schedule a sync when…
  • ✓ A decision requires real-time negotiation between 3+ stakeholders
  • ✓ A project is blocked and async back-and-forth has exceeded 24 hours
  • ✓ Kicking off a new project phase that needs shared mental model
  • ✓ Running a structured design critique that requires facilitation
— Default to async when…
  • — Sharing an update, a file, or a deliverable for review
  • — Asking a question that can wait up to 4 hours for a response
  • — Requesting feedback on a design (use Figma comments + Loom)
  • — Recurring status updates that could be a written post

Expected response times

Urgent / Blocked Tag @here + use 🚨 — within 2 hours
Needs input Same business day
FYI / Review Within 1 business day
DMs Within 1 business day (prefer channels)
After hours No expectation to respond
★
Set your status. When you're in deep work, set your Slack status to 🎧 "Deep work — back at [time]". This signals teammates to use async tools. Respect other people's status the same way you'd want yours respected.

Stakeholder Guide

You don't need to be in every channel or understand every Figma file. This guide is for PMs, engineers, clients, and anyone who works with design but isn't on the design team.

How to give feedback without blocking progress

✓ Do this
  • Leave Figma comments tagged with [FEEDBACK] or [QUESTION] — they're timestamped and tracked
  • Reference the Jira ticket number when raising a concern
  • Watch the Loom walkthrough before giving feedback — context changes everything
  • Consolidate feedback into a single pass — multiple rounds create rework and delay
  • Mark a Figma comment as resolved once your question has been answered
— Avoid this
  • Sending feedback over DM or email — it's not tracked and creates duplicate work
  • Asking for a meeting to share feedback that could be a Figma comment
  • Commenting on aesthetic preferences without tying them to a user or business outcome
  • Reviewing a design without first checking its status in Jira — "READY FOR DEV" work shouldn't receive new feedback
  • @mentioning the whole design team for something one person should own
📌 If you're ever unsure where to direct a question or piece of feedback, post it in #design-announcements or DM the Design Lead on your project. We'd rather answer a routing question than have feedback lost in the wrong channel.

About

The Button is the primary interactive element for triggering actions — submitting forms, opening dialogs, deleting records, and navigating within a flow. It is distinct from the Link component, which is reserved for navigational changes that result in a URL change. Buttons come in four variants that map to a clear visual hierarchy: Primary carries the highest visual weight and should appear once per logical section, while Ghost carries the least and is used for low-emphasis inline actions.

★Semantic rule: Use <button> for actions (submit, delete, open modal). Use <a> for navigation that changes the URL. Never use a <div> or <span> as a button.

When to use / When not to use

✓ Use Primary for

The single most important action in a view — form submission, confirming a purchase, creating a new record. There should be only one Primary button visible at a time in any given section.

✕ Don't use Primary for

Every action on the page. Multiple Primary buttons compete with each other and destroy visual hierarchy. Use Outline or Ghost for secondary and tertiary actions.

✓ Use Secondary for

Brand-colored calls to action in marketing surfaces or landing pages where the yellow treatment aligns with visual intent and sits on a dark or neutral background.

✕ Don't use Secondary for

Actions in dense UI contexts or on light backgrounds where yellow (#FFFF00) has insufficient contrast for text and adjacent color alone. Always check contrast ratios.

✓ Use Outline for

Cancel, Back, and tertiary actions that need to be clearly tappable but shouldn't compete with the Primary action. Pair it with Primary in dialog footer rows.

✕ Don't use Outline for

Actions that carry risk or irreversibility — use Ghost instead, which de-emphasizes the action further and signals lower urgency to the user.

✓ Use Ghost for

Inline or toolbar actions with the least visual emphasis: "Edit", "Clear", icon-only controls in dense rows. Ghost blends with the surface and lets content take focus.

✕ Don't use Button for

Navigation that changes the URL or route — that is the job of the Link component. If clicking it takes the user to a new URL, use an anchor tag, not a button.

Input

Text fields, number fields, text areas, and credit card fields for collecting structured user data in forms and settings panels.

About

Input components collect free-form or structured data from users in forms, settings panels, search interfaces, and data-entry flows. The design system includes four input types: Text Field for single-line text, Number Field for numeric entry, Text Area for multi-line prose, and Credit Card Field for payment flows. All input types share the same visual language — 1px border, --radius-s (4px), and a persistent label above the field.

!Never use placeholder as the sole label. Placeholder text disappears on focus and is not reliably announced by screen readers. Always pair inputs with a visible <label> element.

When to use / When not to use

✓ Use Text Field for

Single-line user input like names, email addresses, URLs, and search queries. It is the default input type for any free-text response that fits on one line.

✕ Don't use Text Field for

Structured selections where a finite set of options exists. Use a Select or Radio Group instead — freeform text increases validation burden and error rates.

✓ Use Text Area for

Multi-line prose input such as descriptions, notes, comments, and feedback. Use when the expected response is likely to exceed one line of text.

✕ Don't use Text Area for

Short structured data like phone numbers, dates, or zip codes. Those need specific input types or masked fields, not a multi-line area.

✓ Use Number Field for

Numeric quantities where the user types a value directly — item counts, ages, prices, or other numeric properties that benefit from increment/decrement controls.

✕ Don't use Number Field for

Phone numbers, credit cards, zip codes, or other "number-like" strings that must retain leading zeros or hyphens. Use a Text Field with appropriate pattern validation.

✓ Do always

Wrap the label and input together in a space-y-2 container so the visual relationship is always clear. Group form sections in space-y-6 for breathing room.

✕ Don't

Place labels to the left of inputs in dense layouts — the label-above pattern is the only approved layout for all Monstarlab product inputs. Left-aligned labels break the system's vertical rhythm.

About

Badges are small, inline status indicators attached to items in tables, lists, cards, and headers. They communicate a semantic state — success, warning, error, or neutral — using color and text together. Each badge variant maps to a specific meaning in the system: use them consistently so users can learn their meaning at a glance. Badges are non-interactive by default and do not respond to click or hover events unless explicitly composed into an interactive element.

!Never use badge colors decoratively. Each variant carries a semantic meaning that users learn over time. Using Success green for a non-success state, for example, breaks the semantic contract and misleads users.

When to use / When not to use

✓ Use Badge for

Communicating the current status of a record in a table row, list item, or card — Active, Pending, Failed, Archived — where a quick visual scan should reveal the state without reading the full row.

✕ Don't use Badge for

Categorization tags that users can filter by or remove. Those have a different interaction model — use the Tag component instead, which supports dismissal and active selection states.

✓ Use Brand for

Highlighting a "New" label, a featured item, or a premium tier designation where the dark background treatment signals special status distinctly from the semantic color variants.

✕ Don't use Badge for

Long text labels or wrapped content. Badges should contain 1–3 words maximum. If the label requires more text, use a different component or truncate with an ellipsis and provide a tooltip.

✓ Do combine Badge with

A data table's status column, a card's header area, or beside a user name to convey role or tier. Positioning should be consistent across all instances of the same entity type.

✕ Don't mix variants

Indiscriminately across pages. If "Active" is displayed as Success (green) on one page, it must be Success on every other page. Inconsistency destroys the semantic value.

✓ Use Default for

Neutral, non-status labels such as category names, version numbers, or archival flags where no semantic color is warranted — just a visual container to draw the eye.

✕ Don't use multiple

Badges on the same item unless they represent genuinely independent dimensions (e.g., status + role). Two or more badges on one row creates visual noise without adding clarity.

Form Controls

Checkbox, radio button, and toggle switch for binary and multi-select decisions in forms and settings panels. All require a visible label.

About

Form controls are the selection primitives of the design system. Checkboxes handle multi-select scenarios where zero or more options can be chosen simultaneously. Radio buttons handle single-select scenarios where exactly one option from a group is valid. Toggle switches handle boolean on/off settings that take effect immediately without requiring a Save action. All three controls share the same active/checked color — #000F1E — and all require a visible label for accessibility.

!A visible label is always required. Never use a form control without an associated label element. A placeholder or tooltip is not a substitute for a persistent visible label.

When to use / When not to use

✓ Use Checkbox for

Multi-select options where the user can independently choose any combination — permissions lists, feature flags, terms and conditions, bulk-select rows in a table.

✕ Don't use Checkbox for

Mutually exclusive choices where only one option is valid at a time. That is the domain of the Radio button group, which communicates the exclusivity through its circular shape and group behavior.

✓ Use Radio for

Single-select choices within a defined, visible set of options — payment method, plan tier, notification frequency. Always show all available options in the group simultaneously.

✕ Don't use Radio for

More than 6 options in a group. When the option set is larger than 6 items, a Select dropdown offers better scanability and uses less vertical space in the form.

✓ Use Toggle for

Boolean settings that take immediate effect: dark mode, email notifications, auto-save, feature toggles. The Switch should live in settings panels, not in forms that require explicit submission.

✕ Don't use Toggle for

Non-binary choices or settings that require a confirmation step before applying. The Switch communicates immediacy — using it for multi-state or deferred settings misleads the user.

✓ Do group controls

Vertically with 12px (space-y-3) gaps in settings panels, or horizontally with 16px gaps for 2–3 short-label radio options. Wrap rich options with descriptions in Card containers.

✕ Don't

Mix checkbox and radio controls inside the same group. The visual language of each communicates selection semantics — mixing them within a single group creates ambiguity about how many items can be chosen.

About

The design system provides three navigation patterns: a Sidebar for primary app-level navigation, Tabs for secondary content switching within a page or section, and a Breadcrumb for hierarchical wayfinding in deep page structures. ✕ Don't use Sidebar for

✓ Use Tabs for

✕ Don't use Tabs for

Icon Button

A square 40×40px button that holds a single icon. Used for toolbar actions and compact interactive triggers where a text label would consume too much space.

About

The Icon Button is a compact, icon-only interactive element for toolbar actions, list-row controls, and inline triggers where space is constrained and the action is visually self-evident from its icon. It differs from the standard Button's icon-only size in that it carries no background or border by default — it relies on the icon itself as the affordance. The Icon Button always requires either a visible tooltip or an aria-label to describe its action to all users.

!Never use an Icon Button without a tooltip or aria-label. An icon alone conveys nothing to screen reader users and may be ambiguous to sighted users unfamiliar with the icon's meaning.

When to use / When not to use

✓ Use Icon Button for

Toolbar actions such as Edit, Delete, Copy, and Share where rows are dense and a text label would disrupt the layout. Use when the icon is universally recognized (pencil = edit, trash = delete).

✕ Don't use Icon Button for

Actions where the meaning is ambiguous without a text label. If you find yourself needing a long tooltip to explain what the button does, use a standard Button with a text label instead.

✓ Use Icon Button for

Repeating actions in list rows or data table cells where vertical space is fixed and a text-label button would overflow the row height.

✕ Don't use Icon Button for

Primary calls to action. The Icon Button's low visual weight makes it unsuitable as the primary action in a view — use a labeled Primary Button for high-priority actions.

✓ Do pair with

A Tooltip component triggered on hover and focus to provide the text label for all users. The Tooltip is the required companion for every Icon Button in production.

✕ Don't group more than

3–4 Icon Buttons in a row without visual separation. Dense icon rows become difficult to scan — add dividers or group related actions into a menu if the toolbar exceeds 4 items.

✓ Do use Lucide React icons

At exactly 18px for all Icon Buttons. Using the correct icon size maintains the optical balance between the icon and the 40×40px touch target.

✕ Don't resize

The touch target below 40×40px. The minimum touch target is 40×40px for desktop and 44×44px for touch-first interfaces. Never reduce the clickable area to match a smaller visual footprint.

Variants

The Icon Button has one structural variant — a 40×40px square with radius 4px. Visual differentiation comes from state, not variant. All Icon Buttons use a Lucide React icon at 18px in #000F1E.

About

Tags are interactive or static labels used to categorize, filter, and identify items with named attributes. !Do not use Tags as status indicators. Tags are for categorization and selection. Use the Badge component for communicating record status (Active, Pending, Error).

When to use / When not to use

✓ Use Tag for

Multi-select filter UI where users choose from a set of categories and can see their active selections. Active Tags visually confirm the filter is applied with the green Active variant.

✕ Don't use Tag for

System status communication. Tags are not status indicators — using Error red or Warning orange Tags to mean "this record failed" creates confusion with the Badge semantic vocabulary.

✓ Use Dismissable Tag for

Selected items in multi-select inputs, applied filter chips, and "pills" that users have added to a field. The close icon signals the item can be removed without navigating away.

✕ Don't use Dismissable Tag for

Read-only taxonomy labels or category displays where removal is not a valid action. A static Tag or Badge is more appropriate when the label cannot be changed by the user.

✓ Use Featured Tag for

Highlighting a specially promoted or pinned item — "Featured", "Recommended", "Staff Pick" — where the dark background treatment creates clear visual separation from standard tags.

✕ Don't mix

Tags and Badges in the same row or cell. Each component carries different semantic meaning — mixing them creates ambiguity about what is a status and what is a category.

Tag groups in a flex-wrap container so they flow naturally onto multiple lines when the viewport is narrow. Set row gap to 6px and column gap to 8px.

✕ Don't truncate

Tooltip

A small floating label that reveals supplementary context on hover and focus. Four positions, dark background, max-width 240px.

About

Tooltips provide brief supplementary context for interactive elements that are visually ambiguous without additional explanation — icon buttons, truncated labels, abbreviations, and disabled controls. They appear on both hover and keyboard focus so they are accessible to all users. Tooltips are non-interactive containers: they must never contain links, buttons, or any other interactive elements, and they disappear when the triggering element loses focus or the pointer moves away.

!Never put interactive content (links, buttons, form controls) inside a Tooltip. Interactive Tooltip content belongs in a Popover instead.

When to use / When not to use

✓ Use Tooltip for

Labeling Icon Buttons where no visible text label exists. The Tooltip is the required label that makes the icon action discoverable and accessible to all users.

✕ Don't use Tooltip for

Essential information that the user must read to complete a task. If the content is required, it must be permanently visible — a Tooltip that disappears hides critical context.

✓ Use Tooltip for

Truncated text labels where the full label is too long to display inline. The Tooltip reveals the complete text on hover so the user can read it without navigating away.

✕ Don't use Tooltip for

Long explanatory paragraphs, error messages, or guidance content. Tooltips max at 240px wide and are designed for 1–2 short sentences. Long content belongs in a Popover or inline help text.

✓ Use Tooltip for

Explaining why a control is disabled — show "You don't have permission to edit this" or "Complete step 1 first" when hovering a disabled button helps users understand what to do next.

✕ Don't use Tooltip on

Touch-only surfaces. Tooltips rely on hover, which does not exist on touch devices. For touch UIs, make the label permanently visible or use a Popover triggered by tap.

✓ Trigger on both

Hover and keyboard focus. Triggering only on hover excludes keyboard users who navigate without a mouse. Both events must show the Tooltip.

✕ Don't add

Interactive elements (links, buttons, checkboxes) inside a Tooltip. If the content requires interaction, use a Popover — Tooltips dismiss on pointer-out and cannot maintain focus.

About

The Link component is the semantic anchor element for navigating to a new URL — whether within the app or to an external resource. Disclosure

An expandable section with a clickable header and hidden content. Used for FAQ sections, settings panels, and progressive disclosure of secondary content.

About

The Disclosure component (also called Accordion) implements the progressive disclosure pattern — showing only the most essential information by default and allowing users to expand sections on demand. This reduces cognitive load in content-heavy interfaces by hiding secondary or advanced content until needed. The Group variant enforces single-open behavior where opening one Disclosure collapses the previously open one, keeping the overall page height predictable.

!Do not hide critical or required information inside a Disclosure. Content that every user needs to complete a task must be permanently visible, not collapsed behind a toggle.

When to use / When not to use

✓ Use Disclosure for

FAQ sections where users have specific questions and only need one answer at a time. The single-open group pattern keeps the overall layout compact while making all answers accessible.

✕ Don't use Disclosure for

Content that most users need to read entirely. If the expanded content is always relevant to all users, display it permanently — hiding it behind a toggle adds an unnecessary interaction step.

✓ Use Disclosure for

Settings panels with multiple categories where most users only need to adjust one category per visit. Collapsed sections reduce visual noise while keeping all categories accessible.

✕ Don't use Disclosure for

Navigation menus that need to stay visible as the user scrolls. Disclosures are content containers, not navigation components. Use the Sidebar's group structure for collapsible navigation.

✓ Use Group for

Related sets of Disclosures (FAQ, settings categories) where the single-open behavior keeps the layout height predictable and helps users focus on one section at a time.

✕ Don't nest

Disclosures inside Disclosures. Two levels of disclosure makes the content structure confusing to navigate and creates an unpredictable interaction pattern for keyboard users.

✓ Do animate

The content expansion with a smooth height transition (200–300ms ease-out). The chevron rotation should animate simultaneously to provide clear visual feedback that the section is opening.

✕ Don't

Default all Disclosures to expanded in a Group. The progressive disclosure pattern works because most content starts collapsed — defaulting all open defeats the purpose and overwhelms the user.

Progress Steps

A step indicator for multi-step flows. Two orientations — horizontal and vertical. Three step states: Completed, Active, and Upcoming.

About

Progress Steps communicate a user's position within a multi-step process and show how many steps remain. They appear in multi-step forms, onboarding flows, checkout processes, and any other sequential task that benefits from visible progress. The component uses three step states — Completed (filled blue with checkmark), Active (outlined blue, bold label), and Upcoming (outlined grey, muted label) — connected by horizontal or vertical lines that fill progressively as steps are completed.

When to use / When not to use

✓ Use Progress Steps for

Multi-step forms and wizards where each step is distinct, sequential, and requires completing before advancing — account setup, checkout flows, onboarding sequences.

✕ Don't use Progress Steps for

Single-step forms or processes with fewer than 3 steps. Progress Steps add navigational complexity that isn't warranted unless there are at least 3 meaningful stages to communicate.

✓ Use Horizontal for

Checkout flows and wizards where the steps are displayed above the form content and the labels are short (1–2 words). Horizontal works well in wide layouts with 3–5 steps.

✕ Don't use Horizontal for

More than 6 steps — they won't fit in a single row at any reasonable size. Switch to Vertical orientation or break the flow into phases for very long sequences.

✓ Use Vertical for

Sidebar-positioned progress indicators, taller forms in narrow layouts, and flows where the step labels are longer descriptions that benefit from more horizontal reading space.

✕ Don't allow

Users to jump forward to incomplete steps by clicking future step indicators. Only completed steps should be clickable (to review or edit). Active and Upcoming steps should not be interactive.

✓ Do show

Step labels below (horizontal) or beside (vertical) each step indicator. Labels provide the semantic meaning of each step — the circle alone is not sufficient to communicate the step's purpose.

✕ Don't

Use progress steps for non-sequential processes where the user can complete steps in any order. The linear visual language implies strict sequence — use a checklist or task list instead.

Popover

A floating overlay triggered by click. Three variants — Simple, WithActions, and Menu. Used for contextual menus, confirmation prompts, and inline help.

About

Popovers are floating panels anchored to a trigger element and opened by click. They are richer than Tooltips — they can contain text, action buttons, and lists — but are focused and contextual, not full modal dialogs. Three structural variants cover the range of use cases: Simple for read-only contextual information, WithActions for confirmation prompts, and Menu for action lists. Popovers close on Escape key or clicking outside the panel.

!Popovers are triggered by click, not hover. If content should appear on hover, use a Tooltip. If content requires the user's full attention and cannot be dismissed by clicking outside, use a Modal Dialog instead.

When to use / When not to use

✓ Use Simple for

Inline contextual help that supplements a form field or UI element — explaining what a setting does, previewing a linked resource, or showing metadata about a record without navigating away.

✕ Don't use Popover for

Critical actions that require the user's undivided attention. If the user must respond before continuing with the app, use a Modal Dialog — Popovers can be accidentally dismissed with a click outside.

✓ Use WithActions for

Lightweight confirmation flows — "Are you sure you want to archive this item?" — where a full modal dialog would be disproportionately heavy for the consequence of the action.

✕ Don't use WithActions for

High-stakes destructive actions like permanent deletion. Those warrant a full Modal Dialog with a deliberate confirmation step (typed confirmation or a clearly labeled irreversible-action warning).

✓ Use Menu for

Context menus and overflow action lists — "Edit", "Duplicate", "Share", "Delete" — attached to a row's kebab menu, a card's options button, or any element with more actions than can be shown inline.

✕ Don't use Tooltip

When the content includes actions, links, or any interactive elements. Tooltip content is purely informational — switch to a Popover whenever the floating panel requires user interaction.

✓ Do close on

Escape key press and click outside the popover. Both dismissal mechanisms are required — keyboard users need Escape and pointer users expect click-outside to close overlay elements.

✕ Don't nest

Popovers inside Popovers. Stacked floating panels create complex focus management problems and confusing z-index layering. Restructure the content to avoid nested floating layers.
