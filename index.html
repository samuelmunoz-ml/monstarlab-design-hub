The Handoff Process

Standard operating procedures for moving designs from Figma to production — eliminating ambiguity between design intent and the final built product.

File Nomenclature & Organisation

A clean, predictable file structure is the foundation of an effective handoff. When engineers open a file, they should instantly understand its purpose, state, and contents — without asking anyone.

Naming Convention

[PROJECT-CODE]_[FEATURE]_[TYPE]_v[VERSION]_[STATUS] e.g. SHR_LoyaltyFlow_UI_v3_HANDOFF.fig  ·  USH_Checkout_UX_v1_REVIEW.fig
TokenDescriptionExample Values
[PROJECT-CODE]2–4 letter client/project abbreviationSHR, USH, MLI
[FEATURE]Feature or module name, PascalCaseLoyaltyFlow, Checkout, Onboarding
[TYPE]File type descriptorUX, UI, Prototype, Assets
[VERSION]Numeric version, zero-paddedv1, v2, v12
[STATUS]Current lifecycle stateWIP, REVIEW, FINAL, HANDOFF

Figma File Page Structure

Every production Figma file must follow this four-page structure in exact order.

🖼

Visualisation of the 4-page structure: Cover · Sandbox · Review · Ready for Dev

Layer & Frame Naming

Auto-generated names like "Frame 47" or "Rectangle 12" are not permitted in the Ready for Dev page.

Element TypeConventionExample
Screen / Frame[ScreenName]/[State]PaymentConfirmation/Success
Component[ComponentName]/[Variant]Button/Primary-Hover
Group[Purpose]-groupnav-icons-group
Text layerDescribe the content roleCTA-Label, Error-Message

★
Pro Tip: Use Figma's built-in layer search (Cmd/Ctrl + F) to audit for unnamed frames before moving anything to Ready for Dev. A zero-tolerance policy on "Frame 47" directly affects how engineers read the Inspect panel.

Definition of Done (DoD)

A design is not done when it looks finished. It is done when it meets every criterion on this checklist. Nothing moves to the Ready for Dev page without passing this gate.

🖼
DoD Checklist — Figma Template
Screenshot of the DoD checklist component as it appears pasted on a Figma frame
A — Coverage & Completeness
  • All primary user flows are fully designed, end-to-end
  • All interactive states present: Default, Hover, Focus, Active, Disabled, Loading
  • All form states covered: Empty, Filled, Validation Error, Success
  • Empty states designed for all data-dependent views
  • Error states designed with specific, actionable copy
  • Responsive variants present for all defined breakpoints (Mobile, Tablet, Desktop)
  • Edge cases documented: long text strings, missing images, maximum data sets
B — Design System Compliance
  • All components are instances from the approved ML Design System library — no detached overrides
  • All colour values map to defined design tokens (no direct hex values on layers)
  • All typography uses defined text styles
  • All spacing follows the defined grid and spacing scale
  • All icons are from the approved icon library at the correct weight
C — Accessibility
  • Colour contrast meets WCAG AA (4.5:1 for normal text, 3:1 for large text and UI components)
  • All interactive elements have visible focus states designed
  • Touch targets meet minimum size requirements (44×44px minimum)
  • All images have alt text or a note indicating decorative intent
  • Reading order is logical and verified via layer panel top-to-bottom review
  • No information is conveyed by colour alone — icons, labels, or patterns supplement
D — File Hygiene
  • All layers in the ready-for-dev page are named per Section 01 conventions
  • Auto Layout applied to all frames used as developer reference
  • No hidden layers in the ready-for-dev page
  • All exported assets included in the designated Exports frame
  • No broken component links in the file
E — Annotation & Documentation
  • All non-obvious interactions annotated with the Annotation component
  • Transition timings and easing curves specified for animated elements
  • Any bespoke or non-standard CSS properties called out explicitly
  • A flow description note added at the top of each flow group

Handoff Status Labels

🔴 BLOCKED 🟡 IN REVIEW 🟢 READY FOR DEV 🔵 IN DEV ⚫ ARCHIVED
LabelMeaningNext Action
🔴 BLOCKEDWaiting on content, assets, or a decisionTag blocker in Jira ticket
🟡 IN REVIEWDesign is complete — awaiting stakeholder or PM approvalPM to approve or comment
🟢 READY FOR DEVAll DoD criteria met — engineers may begin implementationEngineer to accept in Jira
🔵 IN DEVEngineer has accepted and implementation is underwayDesigner to monitor QA

★
Pro Tip: Run the DoD check as a structured async review with a second designer or the PM — not a solo checklist tick. It takes 15 minutes and consistently catches missing edge cases you've become blind to after hours on the same screens.

Access & Permissions

Access to design files must be purposeful and auditable. Every stakeholder should have exactly the access they need — no more.

RoleFigma PermissionScopeGranted By
Design LeadOwnerAll files in the projectDesign Ops Lead
Designer (active)Can EditAssigned project files onlyDesign Lead
Designer (reviewing)Can CommentFiles under active reviewDesign Lead
PM / StakeholderCan CommentReview & Ready for Dev pagesDesign Lead
EngineerCan ViewReady for Dev page onlyDesign Lead
External PartnerView-only linkSpecific shared frames onlyDesign Ops Lead
Source of Truth (SoT) Files
The single approved, versioned file for a project. Only one SoT file exists per project. Always indicated by HANDOFF or FINAL in the filename status token.
Work In Progress (WIP) Files
Exploratory or in-flight files. Never linked from Jira or shared with clients. Prefixed with WIP in the filename and stored in a separate team project folder.
★
External Partner Protocol: Never grant edit access to external partners. Share only the specific frames needed via a view-only link. Revoke access within 48 hours of the review session ending. Log all external shares in the project's Notion page.

Technical Specs & Annotations

Specs bridge the gap between design intent and implementation. Every value an engineer needs must be annotated — if it isn't documented, it doesn't exist.

🖼
Annotation Example — Spacing & Typography Redlines
Screenshot showing a Figma frame with spacing tokens, type styles, and colour tokens annotated

Tokens to Annotate

Spacing Tokens
Use the ML spacing scale (4, 8, 12, 16, 24, 32, 48, 64px). Never annotate arbitrary pixel values — always map to the nearest token and note if a deviation is intentional.
Typography Tokens
Annotate with the text style name (e.g. body-md-regular), not raw values. If a custom size is required, flag it explicitly with a reason.
Colour Tokens
Always reference token names (e.g. surface/primary, text/secondary), not hex values. Direct hex values in annotations indicate a token gap that must be resolved before handoff.
Redlines
Use the ML Annotation component for redlines. Call out padding, margin, gap, and border-radius values explicitly. Annotate the component's responsive behaviour at each breakpoint.

Interactions & Logic

Interactions are the hardest part of a design to communicate in a static file. Every non-obvious behaviour must be annotated — assume the engineer has not seen the prototype.

Required State Coverage

State TypeRequired States
Interactive elementsDefault, Hover, Focus, Active, Disabled, Loading
Form fieldsEmpty, Filled, Focus, Validation Error, Success, Disabled
Data-dependent viewsLoading/Skeleton, Zero data / Empty state, Error / Offline, Populated

Transition Timing Tokens

All animated transitions must be documented using these standard tokens. Format annotations as: transition: [property] [duration]ms [easing-token]

TokenDurationEasingUse Case
transition-instant0ms—Focus ring appearance, state changes where animation would feel janky
transition-fast100msease-outMicro-interactions: hover highlights, button presses, icon state changes
transition-base200msease-in-outStandard UI transitions: colour fades, opacity changes, panel reveals
transition-moderate300msease-in-outModal open/close, dropdown appear, tab panel switch
transition-slow500msease-in-outPage-level transitions, onboarding flows, complex reveals
transition-enter400mscubic-bezier(0.22, 1, 0.36, 1)Elements entering the viewport — slight spring feel
transition-exit250msease-inElements leaving the viewport — faster exit maintains perceived snappiness
★
Pro Tip: When in doubt, be faster. User research consistently shows transitions above 400ms feel sluggish on standard UI elements. Reserve longer durations for motion that communicates meaning — not decoration.

Figma File Hygiene

The structural quality of a Figma file directly determines how efficiently an engineer can inspect, understand, and implement a design. A clean file is a professional act of respect for the people who build what you design.

Auto Layout Rules

Auto Layout is mandatory for all frames intended for developer handoff. It communicates the intended responsive behaviour of a component, not just its static appearance.

Hug Contents
This element should wrap its content — no fixed width/height in the component. Maps to width: fit-content.
Fill Container
This element should be 100% of its parent. Maps to width: 100% or flex: 1.
Fixed
This element has a defined dimension that does not change with content. Maps to an explicit width or height value.
Gap Values
Must reference a spacing token. Padding must be set uniformly or with explicit horizontal/vertical overrides — never set by eye.

Component & Variant Rules

  • Use the ML Design System library for all standard UI elements — buttons, inputs, badges, modals, navigation
  • If a required component doesn't exist, raise a DS request in #design-system before designing a local version
  • Never detach a component to make a visual change — use variant overrides or nested component swaps
  • Variant property names: Title Case, no spaces (e.g. Size, State, Type). Values: lowercase with hyphens (e.g. default, hover, error)
  • Use Figma's Component Property panel — not frame name hacks — for variant values

Inspect-Friendly Practices

  • Turn off "Clip content" on frames that expand with content unless overflow: hidden is intentional
  • Mark all placeholder images as "Fill" with a note specifying the intended aspect ratio and minimum size
  • Remove all unnecessary prototype connections from the ready-for-dev page
  • Add a component description in the Figma component panel for every DS component
  • Zero hidden layers in the ready-for-dev page — hiding is not the same as deleting
★
Pro Tip: Once per sprint, run a quick "Engineer's Review" — open the Inspect panel and try to read your own file as if you're the engineer seeing it for the first time. You'll find annotation gaps, missing states, and ambiguous layout relationships that are invisible from the design view.

Communication Norms

A technically perfect handoff document is only half the equation. A predictable, low-friction communication protocol ensures questions are answered quickly, decisions are documented, and problems are escalated appropriately.

Where Conversations Happen

Communication TypePrimary ChannelSecondary / Escalation
Design questions on a specific frameFigma comment on the relevant frameTag engineer in Jira if no response in 24h
General handoff questions (non-frame-specific)Jira ticket comment on the relevant story#[project-code]-design Slack channel
Urgent blockers (blocking a sprint)#[project-code]-design Slack (tag lead)Direct Slack message to Design Lead
Scope changes that affect designJira ticket — create a linked taskSprint planning meeting agenda item
Design System questions / requests#design-system Slack channelDesign System weekly sync
Client-facing design review feedbackFigma comment (shared link view)PM consolidates into Jira ticket
Design DebtJira ticket tagged design-debtDesign Debt triage — bi-weekly

Figma Comment Protocol

Start every comment with one of these tags, then @mention the relevant person and include enough context that someone reading it three months later will understand it.

[QUESTION] [DECISION] [NOTE] [BUG] [BLOCKED]
  • Resolve comments only when the issue is fully addressed — not when you've replied
  • All comments on a ready-for-dev frame must be resolved before engineers begin implementation
  • Do not use Figma comments for social conversation — use Slack for that
  • Comment threads older than two weeks with no resolution must be escalated to the PM

Handling Design Debt

Design debt is any design decision that is knowingly incomplete, inconsistent, or non-ideal — shipped because of time, scope, or technical constraints. It must be acknowledged, documented, and scheduled — not ignored.

PriorityDefinitionTarget Resolution
P1 — CriticalAccessibility failure or user-facing error state with no resolution pathWithin current sprint
P2 — HighMissing key user flow state or significant inconsistency visible to end usersWithin next two sprints
P3 — LowCosmetic inconsistency, token deviation, or incomplete annotationQuarterly DS review

Debt protocol: Identify → log a Jira ticket with the design-debt label → Annotate the Figma frame with a link to the ticket → Triage bi-weekly with the Design Lead and PM → Resolve and update the Cover page version history.

★
Pro Tip: Never let debt accumulate beyond 15 open tickets without a triage session. When design debt exceeds this threshold, it becomes invisible to the team — no one knows which debt is blocking, which is cosmetic, and which has already been informally resolved. Visibility is discipline.
